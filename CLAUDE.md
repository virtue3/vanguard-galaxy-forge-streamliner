## Project Overview

This is a **BepInEx mod** for the game **Vanguard Galaxy** (a Unity-based space game available on Steam). The mod uses **BepInEx + Harmony** to patch the game at runtime — it does **not** replace or recompile `Assembly-CSharp.dll`. The decompiled game source in `Assembly-CSharp/` is reference material only; it does not need to compile.

The mod DLL is a separate BepInEx plugin that patches specific game methods using `HarmonyLib`.

The mod targets the Steam installation directly. Key paths:
- Game + BepInEx: `E:\STEAM\steamapps\common\Vanguard Galaxy\`
- Game DLLs: `E:\STEAM\steamapps\common\Vanguard Galaxy\VanguardGalaxy_Data\Managed\`
- BepInEx core (includes `0Harmony.dll`): `E:\STEAM\steamapps\common\Vanguard Galaxy\BepInEx\core\`
- Plugin output: `E:\STEAM\steamapps\common\Vanguard Galaxy\BepInEx\plugins\`

The `game\` folder in this repo is an unused leftover — ignore it.

## Build & Deploy

```powershell
cd VanguardMod
dotnet build
```

The post-build step automatically copies `VanguardMod.dll` to the Steam BepInEx plugins folder. Launch the game via Steam normally to test.

BepInEx log: `E:\STEAM\steamapps\common\Vanguard Galaxy\BepInEx\LogOutput.log`

## Architecture

The codebase follows a strict **Data / Behaviour split**:

- **`Source/`** — Pure C# data and logic with no Unity dependencies. This is where game rules, simulation, and data structures live.
- **`Behaviour/`** and **`Behaviour/`** (note: both `Behaviour/` at root and `Assets/_Scripts/Behaviour/`) — Unity `MonoBehaviour` classes that attach to GameObjects and handle rendering, input, and scene interaction.

### Key namespaces and layers

| Namespace | Role |
|---|---|
| `Source.SpaceShip` | Ship data, stats, AI decision data (`SpaceShipData`) |
| `Source.SpaceShip.Auto` | NPC AI action classes (combat, mining, escort, salvage) |
| `Source.Galaxy` | Galaxy/sector/system map data; faction definitions |
| `Source.Simulation` | Game modes via `Storyteller` subclasses (ConquestWorld, SandboxWorld, TutorialWorld) |
| `Source.Simulation.World` | POI management, conquest systems |
| `Source.MissionSystem` | Mission data, objectives, generators |
| `Source.Player` | `GamePlayer` — the root of all serialized player state |
| `Source.Item` | Equipment stats (`EquipStat`), inventory, rarity |
| `Source.Crew` | Crew member data, skill trees |
| `Behaviour.Unit` | `SpaceShip` MonoBehaviour, drones, turrets (Unity side) |
| `Behaviour.Equipment` | Equipment MonoBehaviours (turrets, modules, boosters) |
| `Behaviour.UI` | All in-game UI panels, HUD, popups |
| `Behaviour.Managers` | Singletons: `GameManager`, `LootManager`, `TravelManager`, etc. |
| `Behaviour.Bootstrap` | Scene loading, startup sequence |
| `LightJson` | Embedded JSON library used for all serialization |

### Core patterns

- **Save/Load**: Classes implement `IJsonSource` and serialize via `LightJson`. Entry point is `SaveGame.SaveCurrentState()` / `SaveGame.LoadState()`. `GamePlayer.current` is the root of all save state.
- **Storyteller**: `abstract class Storyteller` controls game mode. Concrete subclasses (`ConquestWorld`, `SandboxWorld`, `TutorialWorld`) define progression caps, world generation, and events.
- **Managers as PersistentSingletons**: Long-lived managers use `PersistentSingleton<T>` (accessed via `.Instance`). Scene-scoped singletons use `Singleton<T>`.
- **NPC AI**: Ship AI is composed from action classes in `Source/SpaceShip/Auto/` (e.g., `CombatActions`, `AmbientMinerActions`, `WingmanActions`). Each action class handles a distinct behavior mode.
- **Equipment builders**: Equipment is procedurally generated via `EquipmentBuilder` using stats (`EquipStat`), aspects, and manufacturer modifiers.
- **Seeded randomness**: `SeededRandom` and `SeedGenerator` are used throughout for reproducible procedural generation.

### Factions

Each faction is a class under `Source/Galaxy/Factions/` (e.g., `Amalgam`, `Fanatics`, `MiningGuild`). Faction identity drives AI behavior, reputation, and mission availability.

### Mission system

Missions are generated by classes in `Source/MissionSystem/Generator/`. Each generator (e.g., `BountyHunt`, `Courier`, `EscortShip`) creates `MissionObjective` chains. Umbral missions are a distinct category under `Generator/Umbral/`.
